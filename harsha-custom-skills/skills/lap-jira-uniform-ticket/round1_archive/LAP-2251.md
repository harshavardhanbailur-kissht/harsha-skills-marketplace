**TL;DR —** Update the internal LOS Issue Tracker's dropdowns. Normally there are **3 dropdowns** (System → Stage → Sub-Category). A **4th dropdown appears only** when the user picks one of three "generic bucket" sub-categories that need more detail: `Tech issue`, `Tech Challenges`, or `Training level issue`. The [taxonomy sheet](https://docs.google.com/spreadsheets/d/1dA3mSKLFoI_OqvDZxRzuPGREUDtm_CXBrYcDzM8wb6A/edit?gid=0#gid=0) is the single source of truth.

## How the form works

Agent fills up to 4 dropdowns. The first 3 are always shown. The 4th only appears when the 3rd dropdown's value is a generic bucket that needs clarification.

### 3 worked examples

**Example 1 — specific sub-category (no 4th dropdown)**
* System = `LOS`
* Stage = `CPA Verified`
* Sub-Category = `Pan DOB is Blank in CAM` → done, submit.

**Example 2 — Tech issue (4th dropdown appears)**
* System = `LOS`
* Stage = `CPA Verified`
* Sub-Category = `Tech issue` → 4th dropdown appears
* Sub-Sub-Category = `Validation Failure` → done, submit.

**Example 3 — Training level issue (4th dropdown appears)**
* System = `LOS`
* Stage = `CPA Verified`
* Sub-Category = `Training level issue` → 4th dropdown appears
* Sub-Sub-Category = `BCM-end issue` → done, submit.

## Cascade rules

1. Picking System updates the Stage list. Changing System resets Stage and Sub-Category.
2. Picking Stage updates the Sub-Category list. Changing Stage resets Sub-Category.
3. Picking Sub-Category: if the value is `Tech issue`, `Tech Challenges`, or `Training level issue`, show the 4th (Sub-Sub-Category) dropdown. Otherwise, hide it — the form is complete.
4. Changing any parent dropdown resets and (if applicable) hides its child.
5. Submit must reject any combination not listed above — server-side validation, not just UI.

```python
# --- Main cascade: System -> Stage -> Sub-Category ---
STAGE_BY_SYSTEM = {
    "LeadGen": ["Interested", "BRE Approved", "Application Onboarded", "Login Fee Collected"],
    "LOS": ["CPA Verified", "In-Principle Approval Pending (IPA)",
            "Third Party Triggers Pending", "Final Sanction Pending",
            "Final Sanction Approval Pending", "Post Sanction",
            "Branch Operations", "Central Operations"],
}

# Conditional 4th dropdown — same 6 options for Tech/Tech-Challenges, distinct 6 for Training
SUBSUB_BY_SUBCATEGORY = {
    "Tech issue": [...6 values...],
    "Tech Challenges": [...same 6 values...],
    "Training level issue": ["CPA-end issue", "BCM-end issue", "SM-end issue",
                             "NCM-end issue", "CCM-end issue", "BOM-end issue"],
}

def needs_sub_sub_category(sub_category):
    return sub_category in SUBSUB_BY_SUBCATEGORY

def on_system_change(system):
    populate("stage", STAGE_BY_SYSTEM[system])
    reset("stage"); reset("sub_category"); hide("sub_sub_category")

def validate(form):
    if form.stage not in STAGE_BY_SYSTEM[form.system]:
        raise ValidationError("stage does not belong to system")
    if form.sub_category not in SUBCATEGORY_BY_STAGE[form.stage]:
        raise ValidationError("sub_category does not belong to stage")
    if needs_sub_sub_category(form.sub_category):
        if not form.sub_sub_category:
            raise ValidationError("sub_sub_category is required")
        if form.sub_sub_category not in SUBSUB_BY_SUBCATEGORY[form.sub_category]:
            raise ValidationError("sub_sub_category does not match sub_category")
    else:
        if form.sub_sub_category:
            raise ValidationError("sub_sub_category must be empty")
```

## Acceptance criteria

- [ ] All dropdowns (System, Stage, Sub-Category, and the conditional Sub-Sub-Category) exist on the issue creation form with exactly the values listed above — nothing more, nothing less.
- [ ] Sub-Sub-Category dropdown is hidden by default and only appears when the user selects Tech issue, Tech Challenges, or Training level issue in Sub-Category.
- [ ] Cascade behaviour works end-to-end in the UI: changing any parent dropdown resets (and hides, where applicable) its children.
- [ ] Server-side validation rejects any combination not in the mapping above. UI-only validation is not sufficient.
- [ ] When Sub-Category is not a bucket that triggers the 4th dropdown, submitting a Sub-Sub-Category value must fail validation.
- [ ] Dropdown values are loaded from a single source (enum / config file / DB seed) so future edits happen in one place.
