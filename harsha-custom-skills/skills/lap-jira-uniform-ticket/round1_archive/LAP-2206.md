**Description:**

> **TAT = Turnaround Time** — the time taken for a case to move through each stage of the loan journey. Used throughout this document.

---

**Objective**

BA to analyze, validate feasibility, and propose LeadSquared-native solutions for the scope areas listed below. Each area should result in a separate child ticket with detailed findings and proposed approach.

---

**Scope Areas**

1. **Case Prioritization & Weighted Visibility**
* Prioritize active cases by a combination of disbursal amount and how long they've been aging
* Higher-value stuck cases should surface first across all views and reports
* BA to validate what loan size bands and weightage logic can be configured in LeadSquared

2. **TAT Breach Tracking — Stage-based**
* Set different TAT thresholds per stage (e.g., login may have a shorter acceptable window than legal vetting)
* Allow different thresholds by product type and loan size
* Track and surface which cases have breached their stage-specific TAT deadline
* BA to propose what stage-TAT combinations are configurable natively vs need custom build

3. **Notifications, Reminders & Escalation to Reporting Manager**
* When a case breaches TAT or has no activity, notify the assigned user
* If the user does not take action after a reminder, send repeated reminders on a defined cycle
* If the user still does not act after multiple reminder cycles, escalate to their reporting manager with full context
* When a query is raised against a user's case or a case is sent back to them, notify them immediately
* All notifications should be consolidated (one per user per day, not per case)
* Multi-channel: in-app first, then email if not viewed, then WhatsApp/SMS as fallback
* _(LAP-1376 has detailed notification templates and acceptance criteria — this epic covers the broader scope)_

4. **Send-back & Query Tracking — Stage, Type, Branch & User level**
* Track queries and send-backs by: which stage they were raised at, what type/category (e.g., income doc mismatch, property doc missing, valuation clarification)
* Branch-level patterns: which branches generate the most queries, and of what type
* User-level patterns (both sides):
    * Which users' cases generate the most queries (indicates uncessary setbacks and increasing TAT)
    * Which users are slowest to resolve queries raised against them (indicates responsiveness)
* Send-back ping-pong detection: same case sent back multiple times for the same reason
* Define expected resolution time per query type (e.g., missing photo = 24h, income mismatch with revised CA cert = 72h)

5. **Branch-wise Segregation & Performance**
* All metrics across every scope area should be sliceable by branch
* Branch performance ranking: disbursal count, disbursal value, average TAT, query volume, first-time-right rate
* Comparative view across branches for central business team

6. **Business Impact Quantification**
* ₹ value of cases stuck at each problem area (TAT breach, query backlog, stagnant pipeline)
* Operational metrics alongside: case counts, rates, trends
* Cost-of-delay: how much disbursal value is being held up and for how long
* This should be visible in the dashboard and summarized in email reports

7. **Rejection Analysis**

_7a. Credit / Internal Rejects:_
* Classify rejection reasons into categories
* Map which stage the rejection happened at vs which earlier stage it could have been caught
* Quantify cost incurred before rejection (third-party expenses like valuation, legal verification, technical inspection)
* Goal: identify where ineligible cases are leaking deep into the funnel and costing money before being weeded out — and whether policy or process changes can catch them earlier

_7b. Customer Drop-offs / Declines:_
* Compile reasons why customers themselves reject or drop off
* Map drop-offs by stage (at what point are we losing interested customers?)
* Detect patterns: are certain branches, products, or loan sizes seeing more customer declines?

8. **Central Business Dashboard**
* Consolidated dashboard for central ops/business team (not branch-level users)
* Should cover: pipeline health, disbursal analytics, bottleneck identification, query hotspots, branch scorecard
* BA to propose what can be built natively in LeadSquared's SIERA reporting vs what needs a custom dashboard

9. **Consolidated Email Reports**
* **Daily exception email:** only surfaces items that need attention — breached cases, stuck queries, escalations pending action. If nothing is wrong, the email should say so briefly.
* **Weekly strategic summary:** branch ranking, TAT trends, query pattern shifts, action items for central team
* **Monthly business review pack:** full-month analytics on disbursals, rejections, TAT, query categories, branch performance — designed to feed directly into monthly review meetings
