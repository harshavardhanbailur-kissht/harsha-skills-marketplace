### Main Applicant Addition:

1. **Sync Mavis + Internal Whitelisted Users (Main + Co-Applicant)**
2. User starts the journey as the normal flow, by entering Name and Phone, Consent and OTP
3. In Mavis, also verify that the user trying to Login for a top-up is a main-applicant, do not allow the co-applicant to start the journey.
4. **Skip Platform level Dedupe -** When the user comes in to login, check if the user is eligible for concurrent + Whitelisted in the Mavis DB.
    1. if yes, do not run the dedupe for such cases and let the case move ahead with OTP Verification along with consent.
    2. If the flag is No, Run the dedupe and continue with the current flow.
5. In the Verify OTP API response for the main applicant, the system will receive an identifier indicating whether the loan is a fresh loan or an internal top-up.
    1. If the user is whitelisted:
        1. Send the flag `is_concurrent_eligible = Yes` to LSQ in the Verify OTP API to indicate that the user is an existing LAP customer with an active LAP loan.
    2. API will share PAN verification status and basic details update status, once both are success only then you will allow user with flag `is_concurrent_eligible` as yes to move ahead in the journey.
    3. Fetch the URN and Basic Details of the Co-Applicant as well. This details will also be shown under the main Applicants' Details Section (Explained further in the flow below)
6. If the user will be from the internal top up whitelisted table the Basic Details should be pre-filled and non-editable in the edit applicant form, except for Marital Status (Email ID?)
    1. LSQ sets the loan type as Internal TopUp and sends the same in basic details update API call.
7. We will then directly take users for adding their address and call the update address API.
8. Delinquency API will be called for all the users.
9. Bureau pull will be triggered; Finbox will also share if user is Top-up eligible 'Yes' or 'No'.
    1. If Top up eligibility is "Null/No" **Reject** the case in case of Top up eligibility
    2. If the eligibility is "Yes" let the case **proceed.**
10. Once Case is BRE approved, Continue with Co-Applicant Addition along with Income related task generation.

---

### Co-Applicant Addition:

1. After the Main applicant is BRE Approved, Allow the user to add co-applicants like normal flow.
2. He should be able to add the same co-applicants as added in the previous case or can add new co-applicants.
3. To know the previously added Co-Applicants, We'll show the basic information related to Co-Applicant such as the Name, Masked Phone Number and Masked PAN under the details section of the main applicant.
4. The checks needed for co-applicants addition:
    1. When the Co-App URN is passed with the main application, make sure we skip the phone dedupe for the co-applicant too and let it pass ahead.
    2. Here, fetch and show co-applicant masked details (so that, if the user wants to add the same co-applicants, he'd know the users)
    3. **If Co-Applicant already exists against the same Main Applicant**:
        * Fetch details similar to Main Applicant (Basic details with marital status being editable) + Present Address
        * Allow to proceed, with the Validation similar to Main Applicant.
    4. **If adding a New Co-Applicant**:
        * Continue with the **current Co-Applicant addition flow (Normal Flow)**
    5. **If Co-Applicant exists against any other case**:
        * **Reject the application**
        * Do not allow to proceed further
    6. **Voter ID Scenario**:
        * If **URN is present but NSDL is not**, **do not allow Co-Applicant to pass**

* Need to skip Aadhaar + Selfie and PAN osv collection tasks for applicant and Co-Applicants if they are added in the previous loan and available in the whitelisted users.
    * If any newly added Co-Applicant Aadhaar + Selfie as well the PAN OSV collection is required as per current flow of addition of the Co-Applicants.
* Once the Application has started for an Applicant (Both Main and Co-App), We should tag the user with a flag that shows it is consumed.

1. Post consumption, meaning the flow has started, the user cannot be added in any other flow is the consumed Flag is yes.
2. If the user is rejected for any reason, the consumed flag will pass No, which will intern allow the user to be added again.
