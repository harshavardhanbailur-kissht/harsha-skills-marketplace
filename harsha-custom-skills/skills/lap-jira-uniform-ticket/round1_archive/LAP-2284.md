## Goal

We want users to get a Google Chat direct message when something important happens on their ticket. Right now we only post to a shared Chat space using a webhook. The webhook stays as is. The new bot adds personal DMs on top.

## Important security rule

The bot must only send DMs to users inside our company Google Workspace domain. Block everything else inside the dispatcher itself. Do not trust upstream filters.

## Concepts to keep clear

* **AssignedTo**: a soft tag on a ticket. Set by the creator or by an admin. Does not mean the person owns the ticket.
* **ClaimedBy**: the admin who has actually picked up the ticket and is working on it.
* **Submitter**: the person who created the ticket.
* **Actor**: the person whose action triggered the event we are about to notify on. Never DM the actor about their own action.
* **Third person on a comment**: a comment author who is neither the submitter nor the current claimed admin.

## Notification rules

1. Ticket created with someone in AssignedTo: DM the assigned person.
2. Ticket claimed: DM the submitter.
3. Ticket released back to the queue: DM the submitter.
4. Ticket reassigned via AssignedTo: DM the new assignee and the submitter. If the assignment is cleared (set to no one), DM the previous assignee that they are no longer tagged.
5. Admin comments: DM the submitter.
6. Submitter comments: DM the claimed admin.
7. Third person comments: DM both the submitter and the claimed admin.
8. Explicit @email mention in a comment: DM the mentioned user. If the same user is already getting a DM from another rule for this same event, send only one DM.
9. Ticket resolved: DM the submitter.

## Cross cutting rules for all of the above

* Never DM the actor of the event.
* Never DM anyone outside the allowed domain.
* Skip duplicates within the same event. One person, one DM per event.
* If a recipient is missing (no claimed admin yet, no assignee, etc.), just skip that recipient quietly.
* A new admin claiming a previously released ticket is not the same as a reassignment. Default behavior is no DM to the previous claimer. Confirm with PM if needed.

## Out of scope

* Removing the existing webhook.
* DMs for normal ticket edits.
* DMs for Jira created updates.
* Slash commands or interactive Chat buttons unless they are actually needed.
* Reopen flow (does not exist today).

## Engineering guiding principles

You choose the design. Just keep these in mind.

* A Google Chat App or Bot is needed. The current webhook cannot do DMs.
* The Chat bot identity may not be the same as the existing Sheets service account. Please run a short spike to confirm what auth, scopes, and Workspace setup are required, and whether a separate service account or Marketplace internal listing is needed. Document what you find.
* Cache each user's DM space resource after the first lookup. Handle cache misses and permission errors cleanly.
* Make DM sending async so it never blocks ticket create, claim, release, resolve, comment, or assignment APIs.
* Add idempotency so the same event firing twice does not send two DMs to the same person.
* Add a small debounce so a fast burst of comments does not become a DM storm.
* Watch for Chat API rate limits. Back off on 429.

## Configuration and safety

* One master feature flag to turn the whole bot off.
* Per rule flags so we can disable a single rule without a deploy if it gets noisy.
* Make sure dev and staging cannot DM real users by mistake. Either point them to a test Workspace or hard disable the bot in non prod.
* A way to mute the bot per user without redeploy.
* Clean structured logs for every DM attempt: recipient, event, ticket, status, latency.
* Recommended: store DM attempts in a small audit collection so we can answer 'did user X get notified for ticket Y' easily.

## Message content

* Keep messages short. Mention the ticket number, what happened, and who did it.
* Add a deep link button that opens the ticket in our app, not the LSQ link.
* Before going live with sensitive fields like TRN, customer description, or LSQ URL inside DMs, please get a quick check from compliance. DMs stay in personal Chat history.

## Acceptance criteria

1. Existing shared webhook still works.
2. DMs are sent according to the rules above. Same person does not get more than one DM for a single event.
3. No DM ever leaves the allowed domain.
4. If DM sending fails, the ticket API call still succeeds.
5. Bot can be turned off fully and per rule from config.
6. Logs clearly show sent, skipped, or failed for each DM attempt.
7. Short doc or README covers the Google Cloud and Workspace setup needed to run the bot.

## Open questions to confirm before development

* Does the existing Google service account work for Chat DMs, or do we need a fresh bot identity and a Marketplace internal app? (Spike)
* Can the bot DM a user who has never interacted with it before, given our Workspace policy? (Spike)
* Do we want the previously claimed admin to be notified when the ticket changes hands?
* Do we want users who were @mentioned earlier in a ticket to also be notified when it gets resolved?

## Notes for the developer

You have room to choose the design. If you find a cleaner way to model these rules inside the existing notification service, go for it. Just do not break the existing webhook flow.
