# 07 — Failure Modes + Recoveries

## Failure mode register

Each row is: failure mode | likelihood | severity | detection signal | mitigation / recovery.

Likelihood: Low / Med / High (relative to a typical week of LAP-team usage).
Severity: Low / Med / High (impact on the downstream release-notes pipeline AND the team's trust in the skill).

## Patterns the failure modes cluster into

Looking at the 18 failure modes, they cluster into 5 mitigation patterns:

1. **Cross-question once, then yield** (F1, F13, F17) — cowork-think-with-me protocol applied directly. The skill never refuses; it questions once and proceeds with PM's choice logged.
2. **Detect contradiction, present both, let PM pick** (F2, F10, F15) — never silent reconciliation.
3. **Hand off cleanly to a sibling skill or flow** (F5, F6, F18) — the skill knows what it's not.
4. **Compaction-resilient state** (F9, F14) — `.lap-ticket-session/` is the contract.
5. **Graceful degradation when external integration breaks** (F4, F11, F12, F16) — local fallback, log for ops, ship usable output.

These five patterns become the skill's **failure-recovery policy**, documented in the skill's reference file `references/failure-recovery.md` so the executor can implement them consistently.

## What's NOT in the failure register (and why)

- **PM provides false information that's later proven wrong** — out of scope. The skill is a faithful capture tool; if the PM says the SLA is 24h and it's actually 48h, the ticket reflects what was said. (Detection of this would require ground-truth knowledge the skill doesn't have.)
- **The Jira API is down** — handled by the underlying MCP server, not by us. Skill exits cleanly with "Jira unreachable, draft saved to markdown only — push when Jira is back."
- **The PM's intent statement is offensive / inappropriate** — out of scope. The skill is for a professional team and assumes good faith.
- **A future kissht-field-release-notes change breaks the contract** — we depend on a stable contract; if it changes, our Phase 7 checks will start failing and we'll update. Versioning of the contract is a maintenance concern, not a runtime failure mode.
