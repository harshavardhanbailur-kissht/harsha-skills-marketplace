## Description

On stage Post sanction, Tab: Post sanction under Subtab: Repayment Details, whenever the user click yes to **'Initiate E-Mandate Registration?'** dropdown from the **Repayment & Disbursal Details Capture - v5** form, the following handling will take place.

* **If the mandate is not approved or first time user does the journey**
    * Link will be shown on the repayment activity.

* **If the mandate is already approved and following conditions are met**
    * Success callback without link will be shared and shown in the opportunity and in the repayment activity.
    * Success callback payload (sample):
        ```
        {
          "enach_session_id": "418f74ba-32bd-433d-92a6-9c966be1aafa",
          "enach_status": "Success",
          "enach_reference_number": "MNDT571757357261DH3K",
          "auth_mode": "debit",
          "juspay_pg_reference": "HDFC7024864030351903",
          "mandate_amount": 650000,
          "mandate_approved_date": "2026-04-06",
          "mandate_expiry_date": "2035-03-07",
          "mandate_frequency": "monthly",
          "status": "APPROVED",
          "user_bank_reference_number": "IBNK177557189752Y7JE"
        }
        ```
    * **Conditions (both must be met):**
        * Sanction Loan amount is `<=` Mandate amount
        * Last EMI date `+ 5 years` `<=` mandate_expiry_date
    * Check the latest NACH Reference number should be sent in the push to LMS api request body and check against the loan in the LMS as well.

* **If the mandate is already approved and the above conditions are NOT met**
    * Link will be shown on the repayment activity.

## Implementation notes

* Conditions are developed in the backend, not in the LSQ.
* Except for `enach_session_id` and `juspay_pg_reference`, every field in the success callback should be mapped in the opportunity and in the repayment activity.

## Edge case — buffer policy

* If the initial mandate expiry is set as 10 years assuming tenure as 5 years and 5 years as buffer.
* On new request for the revised mandate, payment team will check if the tenure / last emi date `+ 4 years` buffer is within the existing mandate expiry — if YES, it will pass the initial mandate details; otherwise it will send a link. **Buffer for the revised mandate is 4 years.**
* If the link is sent on the above scenario, a new mandate will be generated and for this mandate, **buffer should be 5 years.**

**Why this teaches the BO pattern**
1. ONE operator action (`'Initiate E-Mandate Registration?'` → Yes) — the BO trigger requirement.
2. Multiple branches off that action — three primary branches (no mandate / approved-conditions-met / approved-conditions-failed) plus one edge case (buffer policy on revised mandate).
3. Each branch states what the operator sees (link OR success callback) AND the system consequence (push to LMS, callback storage).
4. Mandate buffer policy lives as a named edge case, not buried inside a primary branch — proves the dedup discipline.
5. Backend-only logic explicitly flagged ("Conditions are developed in the backend, not in the LSQ") — this is the "non-overridable rule" pattern the release-notes skill needs to call out as "you cannot change this from the panel".

**Note on the JSON in the body**
This exemplar contains a JSON callback payload — which would normally violate the "no JSON schemas" anti-pattern in `voice-and-style.md`. It's preserved here because the payload is the source of truth for the field mapping rule ("everything except enach_session_id and juspay_pg_reference is mapped"). When writing a new BO ticket with similar payload mapping, prefer a small table (Field | Mapped to opportunity? | Mapped to repayment activity?) over an inline JSON dump.
