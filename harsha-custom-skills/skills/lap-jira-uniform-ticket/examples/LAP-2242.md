The system should route IPA (In-Principle Approval) and FA (Final Approval) decisions to the appropriate approver based on the loan amount slab. Each slab has a designated approver role for IPA rights and FA rights. If the designated approver role is unavailable (not present, or no active user mapped for that role at the applicable hierarchy), the system should automatically fall back to the default approver role for that slab.

## Approval Matrix

| Loan Amount Slab | IPA Approver | FA Approver | Default Approver (Fallback) |
| --- | --- | --- | --- |
| 0 – 7L | CCM | CCM | SCH |
| 7L – 10L | SCH | SCH | NCM |
| 10L and above | SCH | NCM | NCM (IPA) |

## Logic

1. When a case is submitted for IPA or FA, the system should identify the applicable slab based on the loan amount requested.
2. Based on the slab and the approval stage (IPA or FA), the system should route the case to the designated approver role as per the matrix above.
3. **Fallback logic:** If no active user is mapped to the designated approver role for the applicable case (e.g., role is vacant, user is inactive, user is on leave with no delegate set, or no user exists for that role at the applicable branch / region), the case should automatically route to the **Default Approver** for that slab.
4. For the 0–5L slab, where the default is "SCH / NCM," the system should first attempt SCH; if SCH is also unavailable, route to NCM.
5. For the 10L+ slab, "NCM (IPA)" means the user who approved the IPA at the NCM level should be the fallback approver at FA stage — i.e., the system should carry forward the NCM who approved the IPA and route the FA fallback to the same user.
6. The routing decision (primary vs. fallback, and the reason for fallback) should be logged in the case audit trail for every approval action.
7. The matrix should be configurable — business / ops should be able to update slab thresholds, approver roles, and default fallbacks through a config screen without a code deploy.
8. If both the designated approver and the default approver are unavailable, the case should be held in a queue and an alert should be raised to the next level in the hierarchy (or to a central ops pool) for manual assignment.
9. **State-wise SCH variant mapping (SCH 1 / SCH 2)** — Initial mapping:
    * Maharashtra, Andhra Pradesh, Telangana → **SCH 2**
    * Tamil Nadu, Uttar Pradesh, Gujarat → **SCH 1**
10. Production environment mapping reference (see attachments).

## QA Testing Notes

- Approver unavailability is simulated in UAT by inactivating the user at the relevant role / hierarchy level and submitting IPA or FA.
- NCM unavailable behaviour: escalate and raise bug since NCM cannot be made inactive. Please check hierarchy if the correct NCM is present and if not fix that, and if this doesn't resolve it raise it as a bug.
- **Saral journey testing:** Routing decisions are identical between standard and Saral (test default approver here as well).

* **NCM** – National Credit Manager
