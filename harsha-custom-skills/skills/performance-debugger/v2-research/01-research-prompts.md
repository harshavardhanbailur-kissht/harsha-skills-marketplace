## RESEARCH PROMPT 1: Context-Aware Analysis Architecture

You are a senior performance engineering architect designing a context-aware performance analysis system for web applications. The system must detect what tools, frameworks, and intentional design patterns were used to build a project BEFORE suggesting any optimizations.

**Problem statement:** Current regex-based performance scanners flag intentional expensive patterns as bugs. For example, a Three.js project using MeshPhysicalMaterial, 10-effect post-processing stacks, and GSAP proxy animations gets flagged with 15+ "issues" that are actually deliberate premium design choices. Applying these "fixes" destroys the visual quality.

**Research the following:**

1. **Project Type Detection**: How to automatically classify a web project as one of:
   - Static marketing site (mostly HTML/CSS)
   - React SPA (component-based, state management)
   - Three.js/WebGL experience (3D graphics, shaders, post-processing)
   - Animation-heavy site (GSAP, Framer Motion, Lenis smooth scroll)
   - Hybrid (React + Three.js + animations)
   Techniques: package.json analysis, import graph analysis, file extension heuristics, dependency detection.

2. **Intentional Pattern Recognition**: How to distinguish between:
   - "Expensive by design" (MeshPhysicalMaterial chosen for premium look)
   - "Expensive by accident" (MeshPhysicalMaterial used where MeshBasic would suffice)
   Key signals: presence of quality tier system, presence of HDR environment, presence of post-processing, comment annotations, configuration patterns.

3. **Skill Fingerprint Detection**: How to detect if code was generated by specific Claude skills:
   - 3D Web Graphics Mastery patterns (proxy pattern, lagSmoothing(0), delta capping)
   - UI/UX Mastery patterns (Disney animation principles, motion timing variables)
   - Pattern signatures that indicate intentional design decisions

4. **Adaptive Budget System**: How to set different performance budgets based on:
   - Detected project type
   - Target device tiers (ultra/high/medium/low)
   - User-specified priorities (quality-first vs. performance-first)
   - Measured baseline performance (already at 60fps? don't touch it)

5. **Conflict Resolution Protocol**: When the performance analyzer's suggestion conflicts with an intentional pattern, how to:
   - Flag it as "intentional pattern detected"
   - Provide the optimization as an "optional aggressive fix"
   - Explain the quality vs. performance trade-off
   - Never auto-apply destructive optimizations

**Output format:** Design document with detection algorithms (pseudocode), decision trees, and example scenarios showing correct vs. incorrect behavior.
