KEY INSIGHTS FROM RESEARCH
================================================================================

1. NO ONE-SIZE-FITS-ALL SOLUTION
   Different use cases require different refresh strategies:
   - 1s default for balanced use
   - 100ms-200ms for real-time dashboards
   - -1 for batch indexing

2. STREAMING ENABLES INTERACTIVITY
   SSE preferred for 95% of real-time search applications
   WebSocket only needed for bidirectional communication

3. CDC DECOUPLES SYSTEMS
   Debezium + Kafka separates data sources from search indexes
   Enables independent scaling and evolution

4. FST COMPRESSION IS SIGNIFICANT
   Finite State Transducers save 40-60% memory vs tries
   Essential for auto-complete at scale

5. PERFORMANCE BUDGETS PREVENT DRIFT
   Latency creeps up without explicit targets
   P50, P95, P99 should be monitored continuously

6. WEB WORKERS AVOID UI BLOCKING
   SharedArrayBuffer enables zero-copy communication
   Client-side search improves perceived performance

7. INCREMENTAL INDEXING SCALES
   Delta updates faster than rebuilds
   Soft deletes prevent fragmentation without immediate removal

8. EVENTUALLY CONSISTENT IS ACCEPTABLE
   Search indexes don't need strong consistency
   CQRS pattern enables independent read/write scaling

9. MONITORING IS NON-NEGOTIABLE
   Latency, throughput, and health must be tracked
   OpenTelemetry provides standard instrumentation

10. HYBRID APPROACHES WIN
    Local FST + server refresh for auto-complete
    CDC + refresh intervals for balance
    SSE + fallback polling for reliability
