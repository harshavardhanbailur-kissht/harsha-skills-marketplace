# Pipeline Contracts — Handoff Schemas

This document defines the exact data contracts between pipeline stages. Each stage produces
output in a specific format, and the downstream stage (or adapter) expects that format.
When the format doesn't match, an adapter translates.

## Directory Ownership

Each stage owns exactly one directory. No stage reads from or writes to another stage's directory
except through the adapter layer.

```
.deep-think/     ← Owned by: deep-thinker (Stage 1)
.orchestrator/   ← Owned by: orchestrator (Stage 2, optional)
.parallel/       ← Owned by: parallel-builder (Stage 3)
.pipeline/       ← Owned by: super-skill router (metadata only)
```

## Stage 1 Output Contract: `.deep-think/`

The deep-thinker produces up to 8 markdown files. The adapter needs to extract structured
data from these human-readable files.

### Required Files (for adapter consumption)

| File | What the Adapter Extracts |
|------|--------------------------|
| `OVERVIEW.md` | Project name, goal, scope boundaries, success criteria |
| `EXECUTION_CHECKLIST.md` | Ordered task list with verification criteria |
| `ARCHITECTURE.md` | Key decisions, dependency relationships between components |
| `IMPLEMENTATION.md` | File paths, component names, technical specifics |

### Optional Files (enrich but not required)

| File | What It Adds |
|------|-------------|
| `CURRENT_STATE.md` | Existing code context for sub-agent prompts |
| `EDGE_CASES.md` | Constraints and guardrails for sub-agent prompts |
| `OPTIMIZATIONS.md` | Performance targets for verification criteria |
| `CREATIVE_IDEAS.md` | Stretch goals (lowest priority in decomposition) |

### Extraction Patterns

The adapter looks for these structural markers in EXECUTION_CHECKLIST.md:

```markdown
## Phase N: [Phase Name]
- [ ] Task description
  - Verification: [criteria]
  - Files: [file paths involved]
  - Dependencies: [what must complete first]
```

If the checklist doesn't use this exact format (it often won't), the adapter
infers structure from:
1. Numbered sections or phases → become execution layers
2. Bullet points within phases → become individual tasks
3. "After X" / "requires X" / "depends on" language → become dependencies
4. File paths mentioned → become task scope boundaries
5. "Verify" / "test" / "confirm" language → become acceptance criteria

## Stage 2 Output Contract: `.orchestrator/`

The orchestrator produces structured YAML state. This is already machine-parseable.

### Required Files

**`state.yaml`** — The primary state file:
```yaml
project:
  id: "proj_001"
  name: "Project Name"
  goal: "High-level goal"

phase: "implementation"  # planning | implementation | review | complete

tasks:
  - id: "TASK-001"
    name: "Clear imperative action"
    status: "pending"  # pending | in_progress | completed | verified | blocked
    assigned_skill: null  # or skill name
    dependencies:
      hard: []           # Must complete before this starts
      soft: []           # Should complete but not blocking
    acceptance_criteria:
      - "Criterion 1"
      - "Criterion 2"
    outputs:
      - path: "path/to/expected/output"
        description: "What will be created"
```

**`CONTEXT.md`** — Cold-start context for sub-agents:
```markdown
# Project Context
## What We're Building
[goal]
## Key Decisions Made
[architectural decisions]
## Current State
[what exists, what's left]
```

### Extraction for Parallel Builder

The adapter reads state.yaml and:
1. Filters tasks with status "pending" or "in_progress"
2. Builds a dependency DAG from `dependencies.hard`
3. Computes execution layers via topological sort (Kahn's algorithm)
4. Generates sub-agent prompts enriched with CONTEXT.md

## Stage 3 Input Contract: `.parallel/`

The parallel-builder expects these files to be present before execution begins.

### Required: `PLAN.md`

```markdown
# Parallel Execution Plan
**Project**: [name]
**Generated By**: [adapter name]
**Source**: [.deep-think | .orchestrator]
**Complexity**: [SMALL | MEDIUM | COMPLEX]
**Total Tasks**: [N]
**Execution Layers**: [N]

## Layer 0: [Name] — NO DEPENDENCIES (launch all simultaneously)

### TASK-001: [Title]
- **Description**: [What this task accomplishes]
- **Agent Type**: general-purpose
- **Input Context**: [What the sub-agent needs to know]
- **Deliverables**: [Expected output files]
- **Acceptance Criteria**: [How to verify success]
- **Estimated Scope**: [S/M/L]

### TASK-002: [Title]
...

## Layer 1: [Name] — DEPENDS ON Layer 0

### TASK-003: [Title]
- **Depends On**: TASK-001, TASK-002
- **Description**: ...
- **Input Context**: [Include: "Read outputs from TASK-001 and TASK-002 before starting"]
...

## Verification Plan
[How to verify the assembled output]
```

### Required: Directory Structure

```
.parallel/
├── PLAN.md          ← The execution plan (created by adapter)
├── contracts/       ← Interface contracts between tasks (created by adapter)
├── prompts/         ← Individual sub-agent prompts (created during execution)
├── outputs/         ← Sub-agent output files (populated during execution)
└── reports/         ← Verification and merge reports (populated after execution)
```

### Optional: `contracts/`

For tasks with shared interfaces, the adapter creates contract files:

```markdown
# Interface Contract: [Component A] ↔ [Component B]

## Shared Types
[TypeScript/Python type definitions]

## API Surface
[Function signatures, endpoints, props]

## Constraints
[What must be true for integration to work]
```

## Pipeline Metadata: `.pipeline/`

The super-skill router creates this directory to track pipeline state.

### `status.md`

```markdown
# Pipeline Status
**Started**: [timestamp]
**Complexity**: [TRIVIAL | SMALL | MEDIUM | COMPLEX]
**Pipeline Depth**: [1 | 2 | 3]
**Current Stage**: [thinking | orchestrating | parallelizing | complete]

## Stage Log
| Stage | Status | Started | Completed | Files Produced |
|-------|--------|---------|-----------|----------------|
| deep-thinker | complete | ... | ... | 8 files |
| adapter-1 | complete | ... | ... | PLAN.md |
| parallel-builder | in_progress | ... | — | — |

## Decisions
- [Why this complexity level was chosen]
- [Why orchestrator was skipped/included]
```

## Adapter Responsibilities Summary

### Adapter 1: ThinkingCompiler
- **Reads**: `.deep-think/EXECUTION_CHECKLIST.md`, `ARCHITECTURE.md`, `IMPLEMENTATION.md`, `OVERVIEW.md`
- **Produces**: `.parallel/PLAN.md`, `.parallel/contracts/` (if needed)
- **Logic**: Extract tasks → infer dependencies → compute layers → generate prompts

### Adapter 2: StateExtractor
- **Reads**: `.orchestrator/state.yaml`, `.orchestrator/CONTEXT.md`
- **Produces**: `.parallel/PLAN.md`, `.parallel/contracts/` (if needed)
- **Logic**: Parse YAML → filter pending tasks → topological sort → generate prompts

## Error Contracts

When an adapter cannot produce valid output, it creates `.pipeline/ADAPTER_ERROR.md`:

```markdown
# Adapter Error
**Adapter**: [ThinkingCompiler | StateExtractor]
**Source Stage**: [deep-thinker | orchestrator]
**Error Type**: [missing_files | parse_failure | no_tasks_found | circular_dependency]

## What Happened
[Description of the failure]

## What's Missing
[Specific files or data the adapter needed but couldn't find]

## Recovery Options
1. [Re-run the source stage with more detail]
2. [Manually create the missing file]
3. [Skip this adapter and use a different pipeline path]
```
